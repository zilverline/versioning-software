How to
release
software?

Before sharing a change you now must...

  1. Create an issue in Mantis
  2. Create an svn branch (using the issue number)
  3. Commit your changes in the branch
  4. Perform a release via Maven of the branch
  5. Merge the branch back to the trunk

The effect...

It causes quite some overhead,
It gives the assumption you're in control,
It isolates teams, as opposed to share the code base,
And last but not least... merges are prone to errors

There is a simpler way...

Teams should work from trunk, together, on a system that fails fast
triggering communication and shared code ownership

branches are your last resort

Upon a release a tag is created in the versioning system and the artifact is released and stamped with a release identifier

This release identifier is based on 3 numbers:

major.minor.bugfix

bugfix

Upgraded when having internal changes, not changing any interfaces

minor

Upgraded when adding new functionality to a project, extending the API

major

Upgraded when creating a change that is backwards incompatible. This could force clients to change their code.

Let's show an example...

A bug is found in framework version 1.4.0

The fix is released in version 1.4.1

Maven will ask for the next version,

this will be 1.4.2-SNAPSHOT

The result...

  • the trunk contains a POM with version 1.4.2-SNAPSHOT
  • a tag is created identifying version 1.4.1
  • a released version is placed in the internal Maven repository

What do you get?

  • Simpler releases
  • Simple setup in your favorite CI-system
  • Shared code ownership
  • More communication between teams
  • Fast feedback
  • Simpler Maven setup
Will there be challenges?

Sure. Yet better to confront them instead of working around them
